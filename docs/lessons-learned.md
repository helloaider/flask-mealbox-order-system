# 经验教训与复盘

> 复盘格式：**背景 → 当时的决策 → 暴露的问题 → 根本原因 → 解决方案 → 可复用的结论**
>
> 目标是让下一个人（或三个月后的自己）看完就能避开同样的坑，而不是只知道"出了个问题"。

---

## 复盘一：用取餐点开关控制点餐时间——职责错位

### 背景

系统需要支持管理员控制用户能否下单。最直接的想法是：有取餐点才能取餐，所以用取餐点的启用状态来当开关。

### 当时的决策

`PickupPoint.is_active` 全为 `False` 时，后端 `POST /api/orders` 返回 403，前端显示"暂未开放"。管理员通过启用/禁用取餐点来控制点餐窗口。

### 暴露的问题

实际运营时，管理员发现每次开餐前要先去"取餐点管理"把取餐点改为启用，收餐后再改回禁用。取餐点信息（名称、地点、时间段）本身从没变过，却要反复修改状态，操作很别扭。

更大的问题是语义混乱：取餐点被"禁用"，用户以为是这个地点不能用了，而实际是"现在不接单"。

### 根本原因

**职责混乱**。"点餐窗口"是时间维度的概念，"取餐点"是空间维度的概念，二者不是同一件事，不应该用同一个字段控制。

类比：用"餐厅是否营业"来控制"今天午餐是否接单"——餐厅一直在，只是午餐时段才接单，这是两回事。

### 解决方案

引入 `OrderSession`（点餐餐次）表，专门承载时间窗口：

```
order_sessions
├── order_start   开始接单时间
├── order_end     截止接单时间
└── is_active     是否启用该餐次
```

后端下单时检查是否存在当前时间在 `[order_start, order_end]` 内的活跃餐次，没有则拒绝。

取餐点的 `is_active` 只控制是否在用户端展示，与能否下单完全解耦。

### 可复用的结论

> **每个实体只做一件事。** 设计数据模型时，先问自己：这个字段表达的是什么维度的信息？时间、空间、状态、权限——这些维度一旦混用，后期必然出现"改这个会误伤那个"的问题。
>
> **识别信号：** 如果某个操作在业务上说不通（"为什么要禁用取餐点才能关闭点餐？"），说明模型层面的职责划分有问题，不是 UI 问题。

## 复盘二：用户取餐地址用自由文本输入——数据质量失控

### 背景

最初觉得地址就是让用户填个地方，自由文本最灵活，开发最简单。

### 当时的决策

购物车抽屉底部放一个 `<input>` 文本框，用户随便填，字符串直接存 `orders.address`。

### 暴露的问题

管理员后台的订单列表里出现了：

- "1号楼"、"一号楼"、"1号楼门口"、"1楼"——同一个地点四种写法
- 有人写楼号，有人什么都不写直接提交
- 无法按取餐点统计当天各点的订单量

用户侧也有问题：每次都要手动敲，懒得填，或者乱填。

### 根本原因

自由文本无法约束输入格式。当"地址"实际上是一个有限的选项集合时，文本框是错误的控件选择。  
本质上是**数据结构化程度不足**：`address` 是非结构化字符串，既无法筛选，也无法聚合。

### 解决方案

两步走：

第一步，前端改为取餐点下拉选择器，选项来自 `GET /api/menu/pickup`：
- 只有一个取餐点时自动默认选中
- 追加"其他地址（手动填写）"兜底选项

第二步，`orders` 表新增 `pickup_point_id` 外键，下单时双写：

```python
# 选了取餐点
address = pickup_point.name + ' - ' + pickup_point.location  # 文本快照
pickup_point_id = pickup_point.id                             # 结构化 ID

# 手动填写
address = 用户输入
pickup_point_id = None
```

文本快照的意义：即使取餐点日后被删除，历史订单仍能展示当时的地址信息。

### 可复用的结论

> **当"输入"实际上是"选择"时，应该用选择控件。** 判断方法：如果合法的输入值可以事先枚举，就不应该用自由文本框。
>
> **结构化 + 快照双写**是处理"外键可能消失"问题的常用模式。`pickup_point_id` 用于查询和统计，`address` 文本快照用于历史展示，两者各司其职。这个模式在电商订单（商品 ID + 名称快照、地址 ID + 地址文本）里非常普遍。

## 复盘三：时间比较混用 UTC 和本地时间——餐次状态差 8 小时

### 背景

Flask 惯例里 `datetime.utcnow()` 很常见，模型里其他时间字段也用了 UTC，餐次比较时顺手也用了 UTC。

### 当时的决策

```python
def get_status(self):
    now = datetime.utcnow()  # ← 问题所在
    if now < self.order_start:
        return 'upcoming'
```

### 暴露的问题

管理员在后台填写"11:00 开始接单"（北京时间），但 11:00 到了之后状态仍然是 `upcoming`，要到 19:00 才变成 `open`。

### 根本原因

数据库存的是管理员输入的本地时间（北京时间），但比较时用的是 UTC 时间，产生 8 小时偏差。

**根本问题：存入和比较没有用统一的时区基准。**

### 解决方案

统一改为本地时间：

```python
now = datetime.now()  # 与存入的本地时间保持一致
```

这是最小改动的修复方式。代价是系统与服务器时区绑定，换时区会出问题。

更严格的方案（待实施）：
1. 数据库统一存 UTC
2. 前端传时间时带时区偏移，或后端在接收后转换为 UTC 存储
3. 展示时转回用户时区

### 可复用的结论

> **时间处理的铁律：选一种时区基准，全系统贯穿到底，绝不混用。**
>
> 推荐方案：**存储层全用 UTC，展示层按用户时区转换**。这是工业界标准做法。
>
> **识别混用的信号：** 代码里同时出现 `datetime.now()` 和 `datetime.utcnow()`，就要检查它们比较的对象是否来自同一时区基准。

## 复盘四：没有迁移框架，补列逻辑堆进 `app.py`——技术债越来越重

### 背景

体量小，一开始觉得不需要 Alembic，"需要加字段就手写 ALTER TABLE"。

### 当时的决策

在 `app.py` 启动时检测字段是否存在，不存在就补：

```python
if 'session_id' not in cols:
    conn.execute(text('ALTER TABLE orders ADD COLUMN session_id INTEGER ...'))
```

### 暴露的问题

目前已有 5 个补列（`session_id` / `pickup_point_id` / `payment_note` / `combo_types.is_featured` / `users.class_name`），这段代码会无限增长：

- 没有版本号，不知道当前数据库处于哪个"版本"
- 无法回滚
- 每次启动都全量检查字段

### 根本原因

把"数据库变更管理"的责任压给了应用启动逻辑，职责错位。数据库迁移是独立的运维操作，不应该和启动逻辑混在一起。

### 解决方案

当前：维持现状，每次新增补列必须在代码后加注释说明原因和日期：

```python
# 2026-07-02 新增：支持取餐点选择
if 'pickup_point_id' not in cols:
    ...
```

未来扩展时的正确做法：引入 Flask-Migrate（Alembic）：

```bash
flask db init
flask db migrate -m "add pickup_point_id to orders"
flask db upgrade
```

迁移脚本自动生成、有版本号、可回滚、可追溯。

### 可复用的结论

> **"够用就行"的技术决策要记录下来，并设定阈值。** 比如"超过 5 次补列就引入迁移工具"，而不是等到问题严重了才重构。
>
> **判断是否需要迁移工具的标准：** 如果你需要解释"当前数据库是什么状态"，而答案不是"看 models.py 就行"，说明已经需要版本化管理了。

## 复盘五：单文件前端在功能增加后难以维护

### 背景

选择单文件 SPA 是为了零构建、零依赖，方便部署。

### 当时的决策

`frontend/index.html` 和 `admin/index.html` 各为一个文件，所有 CSS、JS 内嵌。

### 暴露的问题

随着功能迭代（加餐次、加取餐点下拉、加搜索），两个文件分别超过 800 行和 1000 行：

- 找一个函数需要大量滚动，没有文件内导航
- 全局变量越来越多（`menuItems`、`combos`、`pickupPoints`、`cart`、`selCombo`、`cachedOrders`……）
- 用户端和管理端的订单卡片渲染逻辑各写了一份，改一处容易忘改另一处

### 根本原因

单文件方案没有模块边界，所有代码在同一作用域堆叠。功能少时感知不到，功能多了代价才显现。

### 解决方案

当前不重构，但明确迭代原则：

1. 新增 JS 函数必须写注释说明用途和参数
2. 全局变量集中声明在文件顶部
3. 相关函数用注释块分组（如 `// ── 取餐点逻辑 ──`）

如果功能继续扩展，最轻量的迁移路径：
- 拆分为 `utils.js`、`cart.js`、`order.js` 等，用 `<script src="">` 引入
- 不需要构建工具，仍然保持零依赖

### 可复用的结论

> **单文件方案的适用边界：功能模块 ≤ 5 个、代码量 ≤ 500 行。** 超出后应该拆文件。
>
> **技术选型时明说局限：** "选单文件是因为简单，但超过 X 行就要拆"这样的约定写下来，比后来被迫重构要好得多。

## 复盘六：API 字段没有前瞻性设计——事后兼容很难看

### 背景

`POST /api/orders` 最初只考虑了"用户填地址字符串"这一种情况。

### 当时的决策

```json
{ "order_type": "custom", "items": [...], "address": "3号楼201" }
```

`address` 直接是字符串。

### 暴露的问题

加入取餐点下拉后，需要同时传 `pickup_point_id`，但原来的 `address` 又不能删（兼容旧数据）。结果后端出现了不优雅的兼容逻辑（两条输入路径并存）：

```python
if pickup_point_id:
    address = pickup_point.name + ' - ' + pickup_point.location
elif address:
    pass  # 直接用文本
else:
    return 400
```

### 根本原因

字段设计没有考虑"地址来源可能多样"这一扩展点，直接定义为字符串，扩展时只能打补丁。

### 解决方案

当前维持兼容逻辑。如果重新设计：

```json
{
  "delivery": {
    "type": "pickup_point",
    "pickup_point_id": 2
  }
}
```

或：

```json
{
  "delivery": {
    "type": "manual",
    "address": "3号楼201"
  }
}
```

`type` 字段明确意图，后续加新配送方式只需扩展枚举，不需要改字段结构。

### 可复用的结论

> **API 字段设计时，识别"可能有多种来源/类型"的字段，用 `type + payload` 结构代替平铺字符串。**
>
> **事后兼容的代价：** 每次兼容老字段都会增加一条代码分支，分支越多，测试矩阵越大。

## 做对的事（值得保持的决策）

### ✅ 价格快照：`OrderItem.price` 在下单时固定

下单时把菜品当前价格复制到 `order_items.price`，不引用 `menu_items.price`。

**为什么对：** 菜品价格会变，但历史订单金额不能变。快照切断了历史数据与当前数据的耦合。  
**可复用场景：** 任何"下单时的状态"都应该快照——价格、库存、规格、地址。

---

### ✅ 软删除菜品：`DELETE` 只改 `is_available`

`DELETE /api/admin/menu/<id>` 只把 `is_available` 置为 `False`，不物理删除。

**为什么对：** `order_items` 外键引用了 `menu_items.id`，物理删除会破坏历史订单展示。  
**可复用场景：** 只要有历史记录引用某实体，就不应该物理删除该实体。

---

### ✅ 自动补列：`app.py` 启动时检测并修复数据库结构

用 `inspect` 检测字段，缺失则 `ALTER TABLE` 补列。

**为什么对：** 保证新版代码在旧数据库上能直接启动，降低非技术用户的升级门槛。  
**使用边界：** 字段数量少、变化频率低的场景。字段多了要换迁移工具。

---

### ✅ 前端 60 秒轮询餐次状态

`setInterval` 每 60 秒重新请求 `/api/menu/session`，自动更新点餐开关。

**为什么对：** 管理员开放餐次后，用户端无需手动刷新。状态变化频率低（分钟级），轮询比 WebSocket 更简单稳定。  
**可复用场景：** 状态变化频率低（分钟级）、实时性要求不高时，定时轮询优于长连接。

---

### ✅ 前端本地搜索：订单过滤不走 API

"我的订单"搜索直接在前端过滤 `cachedOrders` 数组，不发请求。

**为什么对：** 用户历史订单有限（通常几十条），本地过滤比网络请求快，也减少服务器压力。  
**可复用场景：** 数据集小（< 500 条）、搜索频繁、对响应速度敏感时，前端本地过滤优于后端搜索 API。

---

### ✅ 套餐推荐折叠：用数据字段驱动 UI 显隐

套餐是否默认展示由后端字段 `is_featured` 决定，管理员可随时在后台切换，前端无需改代码。

**为什么对：** "显示哪些套餐"是业务配置，不是 UI 逻辑。用数据字段驱动，运营人员可灵活调整。  
**可复用场景：** 任何"哪些项目默认展示/置顶/高亮"的需求，都应该在数据模型里加字段（`is_featured`/`is_pinned`/`sort_order`），而不是在前端写死。

---

### ✅ 餐次与取餐点职责分离

`OrderSession` 管时间（何时可以下单），`PickupPoint` 管空间（送去哪里），两者独立配置、独立控制。

**为什么对：** 职责清晰，管理员操作直观，不会出现"禁用取餐点才能关点餐窗口"的反直觉情况。  
**可复用场景：** 设计实体时先问"这个字段属于哪个维度（时间/空间/状态/权限）？"，不同维度的关注点应由不同实体承载。
